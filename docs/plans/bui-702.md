# BUI-702: prove the free Moksha loop

Source: [Linear BUI-702](https://linear.app/vana-team/issue/BUI-702/verify-money-free-testnet-e2e-after-grant-registration-409-recovery)

Status: active; first real Testnet read proved
Live proof consumer: deployed Career Quest
Smoke/docs owner: `vana-data-app-starter`
Durable path: `docs/plans/bui-702.md`

## What this actually is

We will use deployed Career Quest to make the first real test request, because Vana's own Testnet Apps page links to it and the signed-in user already has the required LinkedIn data:

`Career Quest creates request -> you approve in Vana -> Career Quest reads LinkedIn -> Vana confirms completion`

Then we put the observed path into one repeatable starter smoke command and README. Most of the ticket is observing one run carefully.

Callum does not configure a wallet, add a network, acquire tokens, handle an app private key, or administer a “DCR”. The executor finds and configures the existing team-owned test setup. Callum's only required action is approving the ordinary request in the Vana website.

## Four steps

### 1. Use the verified Testnet setup — ready now

Confirmed live on 14 July 2026:

- **Moksha is Vana's public Testnet.** In a production-hosted app, `?network=moksha` keeps the production Vana website/account surfaces but targets the Moksha chain, gateway, and escrow.
- Vana's Apps page, switched to **Protocol network: Testnet**, links Career Quest at `https://vana-career-coach.vercel.app/?network=moksha`.
- Career Quest forwards `network=moksha` to its request, status, and read routes and requests only `linkedin.profile`.
- The approval page said the user's Personal Server already contained every required LinkedIn data type; no import was needed.
- The first request timed out in Career Quest after the SDK's five-minute approval/readiness window while the Personal Server was still preparing data.
- After the Personal Server became available, a fresh request completed: Vana approved the grant, Career Quest read and rendered the real `linkedin.profile` payload, and Vana returned `status=completed`.
- Career Quest rendered no **Payment receipt** for the successful read. This run settled no Moksha token payment; it did not consume a faucet-funded Testnet balance.
- Vana completed by redirecting to `https://vana-career-coach.vercel.app/connect/return?...&status=completed`, confirming the deployed base URL is `https://vana-career-coach.vercel.app/`.
- The Account Developers “App identities” list is only “saved in this browser”. It is not an authoritative registry lookup.

Two URLs must not be conflated:

- **Launch URL:** `https://vana-career-coach.vercel.app/?network=moksha` — confirmed.
- **Configured base URL (`VANA_APP_URL`):** `https://vana-career-coach.vercel.app/` — confirmed by the completed Vana redirect to `/connect/return`. Successful Moksha grant registration also proves the deployed grantee is registered on the needed network; the browser-saved Account list does not.

Callum has no wallet, token, private-key, or network setup step.

### 2. Complete the Career Quest run and record the first truth — proved

The successful run used a fresh request after the first request exhausted the SDK's five-minute timeout while the Personal Server prepared data. The second approval tab stayed open while Career Quest read.

Then inspect Career Quest:

- A rendered character sheet proved a real mapped LinkedIn payload returned.
- Vana reached `completed`, proving consumer acknowledgement.
- Career Quest already renders a **Payment receipt** section when the SDK settles a fee.
  - This run had no receipt, so this read did not settle a payment or consume Testnet faucet funds.
  - Do not broaden that into “Moksha never charges”; another Personal Server or scope may return a payment challenge.

The Account Testnet funding form says Personal Server reads use native VANA escrow. If this run fails for insufficient escrow, the executor identifies the team-owned Testnet faucet/funding path; Callum is not asked to acquire or bridge anything.

The first failed boundary decides the next move:

| Observation | Inspect first |
| --- | --- |
| Approval fails in Vana | Vana approval surface |
| `/fail` says `grant_registration_conflict` | Gateway/SDK grant registration |
| `/fail` says `missing_external_completion_routing` | `unity-surfaces` routing |
| Approval succeeds but data never becomes readable | Delivery/Personal Server |
| Data is readable but the read fails | Personal Server or SDK fee/read path |
| Read returns an empty/fallback profile | Career Quest mapper or public LinkedIn contract |
| Read succeeds but Vana never reaches `completed` | Consumer acknowledgement boundary |

Stop on the first failure. Fix only its owner, then create a new request.

### 3. Encode the proven path as one starter smoke command — implemented locally

`pnpm smoke:moksha` is implemented on the BUI-702 branch. It:

1. creates a request;
2. prints the Vana approval URL;
3. waits up to eight minutes while Callum approves;
4. polls until Vana says the data is readable;
5. reads exactly once;
6. verifies the returned LinkedIn profile is real rather than the sample/fallback;
7. polls until Vana says `completed`, proving acknowledgement;
8. writes a redacted JSON run record.

The command stores its session cookie itself. It never logs the cookie, private key, full LinkedIn payload, Personal Server URL, or auth headers.

Technical facts the command must respect:

- Vana's canonical `ready_for_read` state currently appears to external SDK consumers as `approved`. Accept either; do not claim the script saw the internal state.
- SDK 3.13.4 discards Vana Web's `/fail.errorCode` and exposes only the lifecycle status. The command cannot distinguish `grant_registration_conflict` from `missing_external_completion_routing` when both arrive as `denied`; it records the broad approval/readiness boundary and requires the Vana tab as separate evidence. Exact automatic ownership needs an SDK/status-contract change.
- The SDK currently swallows acknowledgement errors. Therefore a successful read is not enough: the final `completed` status is the acknowledgement oracle.
- The starter now preserves safe payment metadata through the builder-facing read response so the run can prove whether Moksha charged a testnet asset. Consumer UI does not display it.
- The starter's request binding expires after ten minutes, so the command stops before then.

Local checks:

```bash
mise exec node@24.14.1 -- pnpm test
mise exec node@24.14.1 -- pnpm typecheck
mise exec node@24.14.1 -- pnpm registry:build
mise exec node@24.14.1 -- pnpm exec tsx --test test/registry.test.ts
```

Mocks may test the smoke command itself. They do not complete BUI-702.

One expected oddity: because the smoke command owns the starter cookie and the approval happens in a separate browser tab, the browser's `/connect/return` page may say the request is unavailable. That page is not the oracle for this scripted run; the command's signed status check is.

### 4. Document and repeat from clean checkout — author docs added; stranger run pending

After one successful run:

- Put the exact prerequisites and `pnpm smoke:moksha` command in the starter README. **Done locally.**
- Document the observed Moksha fee/faucet fact, not an assumption. **Done locally.**
- Have someone who did not build the lane follow only the README from a clean checkout. **Blocked until the team-owned Moksha identity's approved secret-sharing owner/location is named.**
- Any verbal rescue becomes a README bug; fix it and rerun.

## Run record

Each run writes `.scratch/bui-702/runs/<timestamp>.json` with:

- starter commit SHA, SDK version, and SDK release-tag commit SHA (or an explicit `unknown` provenance gap);
- request ID and safe approval URL metadata;
- timestamped external statuses;
- safe HTTP error kinds;
- returned scope, mapped field names, and payload hash—but no payload;
- safe Moksha payment fields: asset, amount, fee breakdown, paid-at time;
- final status;
- first failed boundary and proposed owner.

Screenshots and `/fail` evidence are referenced separately and redacted.

## Done

BUI-702 is done when:

- an author run reaches `completed` on Moksha;
- a README-only clean-checkout run by another person also reaches `completed`;
- both runs prove the fee path and contain redacted evidence;
- the starter README contains the exact proven workflow;
- tests, typecheck, build, and registry parity pass;
- any cross-owner fix is proved in its own repository.

## Terms, once

- **Moksha:** Vana's test network; no real money.
- **App identity:** a server-side credential used by the starter, not Callum's wallet.
- **Test user:** the ordinary Vana login used to approve the request.
- **Personal Server:** the Vana surface that serves the user's approved data.
- **DCR / data connection request:** one permission-and-delivery job. A new run creates a fresh request ID.
